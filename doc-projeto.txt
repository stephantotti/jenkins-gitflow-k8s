***CRIAÇÃO DO CLUSTER***

Criar o Cluster através do arquivo de configuração "cluster-config.yaml"
 - kind create cluster --config cluster-config.yaml
 - kubectl get pods -A // Para testar a conexão com o cluster e ver se está tudo runnning

* O kind inicia 3 containers que simulam os nodes no ambiente, e dentro desses nodes haverão pods com containers, ou seja, é o conceito de docker in docker.

docker stop $(docker ps -q)
  - Para parar os containers e não ficar consumindo recursos computacionais.

docker start kind-control-plane kind-worker kind-worker2
  - Inicia os containers parados 

***CONFIGURAÇÃO DO HOSTS***

docker ps 
 - Lista os containers
Escolhe o ID de um containers

docker inspect "id do container"
 - Acha o IPAdress // Ex: 172.18.0.3
 - Escolhe um endereço IP nesse range para ser o Load Balancer // Ex: 172.18.0.50

Cria o yaml do DaemonSet
  - kubectl apply -f setup-hosts-daemonset.yaml 
  - Configura o IP do load balancer no manifest 

A idéia do DaemonSet é para ter um pod em cada node contendo um init container para fazer a configuração automatica dos hosts.

Após subir o DaemonSet, use docker exec -it "id de um container" bash e de um cat no arquivo /etc/hosts pra ver se a configuração está correta!

***INSTALAÇÃO DO METALLB*** Muito utilizado em ambientes On Premisses // é um add on para o kubernetes que permite que a gente tenha um endpoint do tipo LoadBalancer (endereço IP)

* Necessário verificar se utiliza kube-proxy em IPVS mode, se sim, é preciso fazer uma configuração prévia antes.

kubectl get cm -n kube-system kube-proxy -o yaml
  - Procurar o parametro "mode"

No meu caso estava IPTables, então não há necessidade de realizar a pré configuração.

INSTALAÇÃO
kubectl apply -f https://raw.githubusercontent.com/metallb/metallb/v0.15.3/config/manifests/metallb-native.yaml

Necessário executar este comando para que se trave o terminal, até que todos os pods com a label --selector=app=metallb estejam com a condition de Ready, ou seja, o metallb esteja completamente instalado
@kubectl wait --namespace metallb-system \
	  --for=condition=ready pod \
	  --selector=app=metallb \
	  --timeout=300s

***CONFIGURAÇÃO DO METALLB*** Encontrar a faixa de endereços IPs dentro da rede do cluster, para distribuir para os services do tipo Load Balancer.
docker inspect kind | jq -r '.[].IPAM.Config[1].Subnet' // No meu caso, é o json path correto, pode ser que em outros caso devolva um IPV6 e seja necessário configurar

***DEPLOY NGINX INGRESS CONTROLLER*** O Pod do nginx ingress controler vai estar exposto para a "internet/nossa máquina" via um service LoadBalancer

Instalar helm
  - sudo snap install helm --classic

Instalar nginx ingress controller
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update
helm search repo nginx // Para procurar o chart instalado

Criar o arquivo "values" do ingress-nginx
helm template nginx ingress-nginx/ingress-nginx -f values/ingress-nginx/values.yaml // Pega as informações do seu nginx e joga no arquivo values

helm upgrade --install --namespace ingress-nginx --create-namespace -f values/ingress-nginx/values.yaml ingress-nginx ingress-nginx/ingress-nginx // Realiza a instalação de fato

Após isso será gerado um service do tipo LoadBalancer com o External IP que foi definido e é possível testar a conexão
curl -Lvk 172.18.0.50

Instalar o HelmFile
  - Necessário baixar o binário no site oficial e realizar a instalação
tar xvzf helmfile_1.4.4_linux_amd64.tar.gz 
sudo chmod +x /usr/local/bin/helmfile

***IMPORTANTE***

EM TODOS OS VALUES HÁ CREDENCIAIS, NÃO COMITTAR PARA O REPOSITÓRIO REMOTO!!!

Para todos os "values.yaml" criados, a configuração está no repositório do github de cada ferramenta
Criar os values.yaml para cada ferramenta

Configurar o HelmFile com as informações do chart do ***ingress-nginx***
Todas as infos para preencher os parametros estão no comando abaixo
helm upgrade --install --namespace ingress-nginx --create-namespace -f values/ingress-nginx/values.yaml ingress-nginx=nome da release ingress-nginx/ingress-nginx=repo/chart

PARA TODAS AS FERRAMENTAS ABAIXO!
Adicionar o ip do load balancer e as urls ao arquivo /etc/hosts/ ***Se for WSL não vai funcionar o ip do load balancer, será necessário utilizar 172.18.0.1 e adicionar no hosts do windows***
Todas as configs abaixo estão na aula denominada "resumo" do curso

***COM TODOS OS SERVIÇOS UP***
1 - Entrar no Gitea e criar uma nova organization (privada)
2 - Criar o repositório abaixo da organização
3 - Criar uma chave ssh
  -  cd $HOME/.ssh // Lista as chaves existentes
  -  ssh-keygen // cria a chave ssh // ***A CHAVE PRIVADA NUNCA SAI DA SUA MÁQUINA, ELA É USADA PARA CLONAR REPOSITÓRIOS, NO REPOSITÓRIO REMOTO SE USA A PÚBLICA!***
  -  cat nome da chave pública criada
  -  copiar e colar em setting > manage ssh keys > add ssh key

4 - Criar um usuário de serviço e dar acesso a ele ao repositório (usuários de serviço usam ferramentas especificas pra eles, ex: jenkins)
  - Site administration > identity & access > user accounts > create user accounts
  - Criar um time de devops, adicionar o user criado como membro, e fazer conexão ssh (gerar uma chave nova pra ele), para poder clonar o repositório após expor a porta ssh 22

5 - Expor o gitea via ssh // gitea é um SCM = Source Code Manager
  - Alterar o values do ingress-nginx:
tcp:
    22: "gitea/gitea-ssh:22"

Após isso, basta clonar o repositório e dar um ls pra ver se foi concluído o processo!
Necessário clonar o repositório dentro do container tamb´me, da mesma forma.

***CONFIGURAÇÃO DO WEBHOOK*** // Mecanismo de comunicação entre sistemas onde um servidor envia automaticamente uma notificação HTTP para outro sistema no momento que um trigger acontece.
- Criar uma multi-branch pipeline no jenkins
- Escolher a opção scan by Webhook
- Configurar o webhook e a URL, http://jenkins.jenkins.svc.cluster.local:8080/multibranch-webhook-trigger/invoke?token=restapi-flask (tem na documentação do plug in multibranch scan webhook trigger)
- Verificar os logs pra ver se o webhook funcionou // kubectl logs "nome do pod" -f -n jenkins
- Foi necessário instalar o ngrok para o webhook funcionar sem um túnel aberto em outro terminal // Kind não tem essa propriedade
